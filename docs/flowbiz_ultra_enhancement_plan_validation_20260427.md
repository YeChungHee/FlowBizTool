## 1. 종합 판정

고도화 계획서의 내용 방향은 현재 코드 검증 결과와 대체로 일치한다. P0/P1/P2 구분, SourceQuality 도입, active framework/promote 경로, CLI/웹 학습 기준 통합, evidence/confidence 기반 평가 고도화, pytest 기반 회귀검증 계획은 모두 타당하다.

다만 원본 DOCX 파일은 문서 구조 품질상 바로 최종 배포하기 어렵다. `Heading1`, `Heading2`, `Heading3` 스타일 ID가 중복되어 artifact-tool 렌더링이 실패했고, 임시로 중복 스타일을 제거해 렌더링했을 때 여러 페이지에서 표 내용이 지나치게 좁은 열에 들어가 세로로 쪼개져 보였다. 또한 표 8개는 첫 행이 반복/헤더 행으로 지정되어 있지 않아 접근성 감사에서 중간 등급 이슈로 잡혔다.

결론적으로 계획서의 기획 내용은 승인 가능하지만, DOCX 형식은 스타일 중복 제거와 표 레이아웃 재정비 후 다시 렌더 검증해야 한다.

## 2. 내용 검증

### 2.1 검증 보고서 반영 여부

계획서에는 이전 전체 검증 보고서의 핵심 진단이 반영되어 있다.

- 학습 케이스 7건, 업데이트 적격 5건, 누적 weight 4.30, 업데이트 조건 미충족
- 상담보고서 2건 본문 추출 성공, 심사보고서 2건 Notion integration 권한 오류
- 업데이트 산출물이 실제 평가엔진에 자동 적용되지 않는 문제
- 자료 품질이 아니라 문자열 존재 여부로 학습 적격을 계산하는 문제
- CLI 학습 루프와 웹 학습 루프의 적격 기준 불일치
- 문서 추출 결과가 세부 subfactor로 충분히 매핑되지 않는 문제
- 결과 검증 및 테스트 체계 부족

### 2.2 고도화 로드맵 타당성

4단계 로드맵은 위험도 순서가 맞다.

1. 학습 적격 게이트 정비
2. 평가 입력 변환 고도화
3. 업데이트 적용 경로 구축
4. 결과 검증 자동화

특히 1단계에서 `SourceQuality`를 먼저 도입하는 판단은 적절하다. 현재 가장 큰 운영 리스크는 파싱 실패 자료도 업데이트 적격으로 인정될 수 있다는 점이기 때문이다.

### 2.3 보완이 필요한 내용

계획서에 추가하면 좋은 항목은 다음과 같다.

- 1단계 완료 후 기존 7건 registry의 재계산 전후 비교표를 산출물로 명시한다.
- SourceQuality 기준에 `usable_for_evaluation`과 `usable_for_update`를 분리한다. 상담보고서는 평가 반영에는 쓰되, 업데이트 반영은 더 엄격하게 보는 구조가 필요하다.
- Notion 권한 오류 해결은 수동 조치뿐 아니라 API 응답상 401/403/404를 UI에 명확히 표시하는 자동 경고까지 포함한다.
- promote 기준에 최소 안정성 조건을 명시한다. 예: 기존 케이스 decision 변화율, 한도 변화율, P0 오류 0건.
- 회귀 테스트는 pytest뿐 아니라 최소 CLI 검증 명령도 정의한다. 예: `python3 -m py_compile ...`, `python3 bizaipro_learning.py status`, sample evaluate.

## 3. DOCX 구조 검증

### 3.1 렌더링 실패

원본 DOCX를 artifact-tool로 렌더링했을 때 다음 오류로 실패했다.

- 중복 style id: `Heading1`, `Heading2`, `Heading3`
- 오류 요지: `Argument_AddingDuplicateWithKey, Heading1`

원인:

- `word/styles.xml` 안에 동일 styleId가 중복 정의되어 있다.
- 확인 결과 style 수는 19개이고, 중복 styleId는 `Heading1`, `Heading2`, `Heading3` 각각 2개다.

영향:

- 일부 DOCX 처리기와 자동 렌더러에서 문서가 열리지 않거나 변환 실패할 수 있다.
- 향후 자동 검증, PDF 변환, 문서 병합, 스타일 정규화 작업에서 오류가 반복될 가능성이 높다.

### 3.2 임시 복구본 렌더링 결과

원본은 수정하지 않고, 중복 스타일만 제거한 임시본을 만들어 렌더링했다.

- 렌더 결과: 28페이지
- 페이지 크기: 1414 x 2000 PNG
- 렌더 자체는 성공

시각 확인 결과:

- 일부 표/리스크 박스가 지나치게 좁은 열에 배치되어 글자가 세로로 쪼개진다.
- 목차 이후 빈 페이지 또는 내용 밀도가 매우 낮은 페이지가 다수 보인다.
- 14개 표 중 일부가 페이지 폭과 내용량에 비해 과도하게 분할되어 문서 가독성을 떨어뜨린다.

### 3.3 접근성 감사 결과

모든 medium 이슈는 `table_no_header_row`다. 표 첫 행에 `w:tblHeader`가 없어 다중 페이지 표 또는 보조기기 읽기에서 헤더 행 인식이 약해진다.

## 4. 수정 권고

### 4.1 내용 수정

내용은 대체로 유지하되 다음 문장을 추가하는 것이 좋다.

- `SourceQuality`는 평가용과 업데이트용 기준을 분리한다.
- 기존 registry 7건은 정규화 전/후 결과를 비교 리포트로 저장한다.
- Notion 권한 오류는 UI에서 “자료 있음”이 아니라 “본문 미확인”으로 표시한다.
- promote 기준에는 decision 변화율과 한도 변화율 상한을 둔다.
- 테스트 명령과 산출물 경로를 각 단계별 완료 기준에 넣는다.

### 4.2 문서 형식 수정

원본 DOCX를 최종 배포용으로 쓰려면 다음 수정이 필요하다.

1. 중복 스타일 ID 제거: `Heading1`, `Heading2`, `Heading3`
2. 표 첫 행 헤더 지정: 주요 8개 이상 표에 `w:tblHeader` 적용
3. 리스크 분석 섹션의 2열 표 재구성: 우선순위 배지는 좁게 유지하되 본문 설명은 별도 문단 또는 넓은 단일 셀로 배치
4. 목차와 빈 페이지 정리: 내용 없는 페이지 제거
5. 렌더 후 전 페이지 시각 검수: 세로 글자, 잘림, 표 압축 여부 재확인
